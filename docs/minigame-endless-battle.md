# 付属ミニゲーム提案「無限回廊」

本編（フローチャート型パズル RPG）に付属させる、スコアアタック型ミニゲームの
設計案をまとめる。

## 狙い（要件）

ユーザーから提示された 3 要件：

1. **一人でやるゲーム**であること
2. 最終的に**スコアを出して他ユーザーと競える**こと
3. **中毒性があり何度もプレイしたくなる**こと（1 プレイ 5 分以内）

## 全体コンセプト

**「無限回廊」** — 本編の戦闘エンジンをほぼそのまま流用したエンドレス連戦の
スコアアタック（ローグライト）。HP を引き継ぎながらウェーブを連続で突破し、
「何ウェーブ到達できたか」を競う。

- 敵・カード・戦闘ロジック・ピクセルアートを**全再利用**でき、新規アセットが
  ほぼ不要 → 実装コストが最小。
- 手札と盤面が毎回変わるため、同じ攻略が二度と成立せず反復性が高い。
- HP 引き継ぎの緊張感で「あと 1 ウェーブ」が止まらなくなる。
- 1 プレイは HP が尽きた時点で自然に終わり、数分に収まる。

## 「詰み」問題とその解決

本編のフローチャートは「答えが一通り」になるよう設計され、詰まないことを
保証していた。エンドレスでは盤面をランダム生成するため、この保証を別の形で
担保する必要がある。

### 2 種類の「詰み」を区別する

| 種類 | 中身 | 扱い |
|---|---|---|
| フェアな敗北 | HP を使い切って run が終わる。置き方が甘かった／引き継ぎ HP 不足 | **仕様**。スコアが確定して次の挑戦へ。中毒ループの燃料 |
| アンフェアな詰み | どう置いても勝てない盤面・手札が配られる。技量と無関係に負ける | **潰すべきバグ**。理不尽で離脱を招く |

エンドレスの設計契約：**答えは複数あってよいが、「勝てる置き方が最低 1 つは
必ず存在する」ことを保証する。**

### 解決手段：既存エンジンを神託にした生成→検証

`frontend/src/engine/simulateBattle.js` は、アニメ・遅延抜きで盤面を最後まで
数値計算し `win` / `lose` / `runaway`（無限ループ）を即判定する純関数。本編が
実行前に勝敗を先読みするための仕組みだが、これを**そのまま「詰み判定ソルバー」**
として使える。

生成時のアルゴリズム（rejection sampling）：

1. ランダムに盤面（敵 HP・フローチャートの形・手札）を生成する。
2. **手札をスロットに入れる全パターンを列挙**し、各パターンを `simulateBattle`
   にかける。
3. `win` になる置き方が 1 つも無ければ盤面を捨てて再生成する（または敵 HP を
   下げる／攻撃カードを 1 枚足す）。

スロット数もカード枚数も少ない（概ね 1〜5）ため順列は数十〜数百通りで、生成時
に総当たりしても一瞬。新しいソルバーを書かずに既存エンジンを oracle にできる。
`runaway` 判定込みなので無限ループに落ちる盤面も自動で弾ける。

### 引き継ぎ HP に絡む、もう一段深い詰み

engine の仕様上、**guard のシールドと reflect は「直後の 1 ノードだけ」有効**
（`clearTransientBuffs`）。locked な `monster` カード＝敵の攻撃で、その直前に
guard / reflect を置けないと必ず食らう。

そのため「勝てる」だけでは不十分で、低 HP 時に「勝てるが最低 ◯ ダメージは
避けられない盤面」が来ると理不尽な詰みになる。保証を一段強める：

> **各ウェーブは「勝てる置き方のうち被ダメージ最小の line」のコストが上限 C
> 以下であることを保証する。**

- 序盤：C = 0。guard / reflect を正しく置けば**無傷で勝てる line** が必ず存在。
- 中盤以降：C を少しずつ上げ、「多少削られるのは避けられない」緊張を作る。
- C の算出も `simulateBattle` を各置き方に回し「勝てる line の最小被ダメ」を
  取るだけ。

これで敗北は必ず「HP バッファ管理をミスった／リスクを取った結果」になり、
運ゲーにならない。

## ゲーム設計（確定事項）

| 項目 | 決定 |
|---|---|
| ゲーム性 | 一人用エンドレス連戦スコアアタック（戦闘エンジン・敵・カードを全流用） |
| 手札 | 毎ウェーブ ランダム配布 |
| 詰み対策 | `simulateBattle` を神託にした生成→検証。勝てる置き方が無い盤面は再生成 |
| 被ダメ保証 | 各ウェーブ「勝てる line の最小被ダメ ≤ 上限 C」。序盤 C=0 → 進むほど C↑ |
| HP | 全引き継ぎ・ウェーブ間の自動回復なし（heal カードが貴重資源になる） |
| 逃げ道 | **回数制限つきのカード・リロール**（手札だけ引き直す／ウェーブは進めない） |
| 難易度カーブ | 敵 HP↑ → monster ノード（被ダメ）増 → 手札の余り枚数↓ → 分岐/条件つきの複雑な形、の順で効かせる。ゆるく始め中盤から急峻に |
| フローチャート | 形はテンプレ集からランダム選択し、敵/カード/HP を乱数で埋める。完全自動生成よりテンプレ＋乱数が安全・多彩。生成後に必ず検証 |
| スコア | **到達ウェーブ数のみ**（指標一本でランキング比較が容易） |
| ランキング | まず localStorage 自己ベスト → 将来 Python サーバーで全体ランキング |

### 逃げ道を「リロールのみ」にした理由

スコアを到達ウェーブ数だけにすると、「スキップ＝戦わずに次ウェーブへ」は
タダでスコアを稼ぐ抜け道になる。リロールは**同じウェーブの手札を引き直すだけ**
でウェーブを進めないため、スコアに影響しない。生成が「必ず勝てる」を保証して
いるので、逃げ道の役割（見えづらい／HP を食う手札の引き直し）はリロールだけで
十分果たせる。スキップは採用しない。

## 検討したが採用しなかった案

- **B 案「一手詰めパズル」**：固定手札＋盤面で最大ダメージの並べ方を探す瞬間
  パズル。じっくり考える系。日替わり問題向き。→ 反復性より思考寄りで今回の狙い
  と外れる。
- **C 案「60 秒タイムアタック」**：制限時間内に何体倒せるか。反射神経・爽快感
  重視。→ せわしなく、本編のパズル性を活かしにくい。
- **手札：デッキ構築型（ローグライク）**：戦ごとにカードを 1 枚選んで山札を育てる
  Slay the Spire 風。「自分で選んだ感」は大きいが実装が重い。→ まずは毎ウェーブ
  ランダム手札で最小構成。
