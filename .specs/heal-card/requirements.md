# 要件定義: ヒールカード処理（HP回復と演出）

## はじめに

フローチャート実行中に `heal` カードが配置されたスロットを通過した際、その `card.power` 値分だけプレイヤー HP を回復する戦闘ロジックを実装する。あわせて「回復した」ことを伝える演出（プレイヤー HP バー周辺の緑系フラッシュ・回復数字「+<amount>」のフロート表示・HP バーの滑らかな増加）を加える。

`heal` カードは既に `stages.json` の各ステージ `cards` 配列に定義され（例：`{ "id": "heal", "power": 12 }`）、画像 `public/cards/heal.png` も用意されている。手札からのドラッグ&ドロップ・スロットへの配置といった UI 側の振る舞いは attack / guard と同じ規約で既に成立しているため、本スペックではフローチャート実行時の **回復ロジックと演出** のみを対象とする。

プレイヤー HP の状態管理・実行ごとの初期化・動的 HP バー・被弾ダメージのフロート表示は `monster-attack` スペックで既に実装済みであり、本スペックはそれを前提として「回復方向（HP 増加）の処理と演出」を追加する位置づけ。対象利用者はステージをプレイするプレイヤー本人。

## 要件

### 要件1: ヒールカードによるHP回復の適用

**ユーザーストーリー：** プレイヤーとして、フローチャート実行中に heal カードを配置したスロットを通ったとき、そのカードの `power` 値だけ自分の HP が回復してほしい。なぜなら配置したカードの値が結果に直結することで、「どのカードを置けばいくつ回復するか」というプログラミング的因果関係を学べるから（attack カードによるダメージ適用と対称の挙動）。

#### 受け入れ基準

1. WHEN 実行シーケンス進行中にスロットノードがハイライトフェーズに入る AND そのスロットに `id === 'heal'` のカードが配置されている THEN the system SHALL そのカードの `power` 値をプレイヤー HP に加算する
2. IF 配置されているカードが `heal` 以外（`attack` / `guard` / `monster` 等）の場合 WHEN スロットノードがハイライトフェーズに入る THEN the system SHALL プレイヤー HP は変化させない
3. WHEN 回復が適用される THEN the system SHALL ハイライトフェーズの開始タイミングに合わせて加算を実行し、緑フラッシュ演出・回復数字フロート・HP バー増加の各演出を時系列で同期させる
4. IF `power` が 0 または欠損している WHEN heal カードのスロットがハイライトフェーズに入る THEN the system SHALL 回復適用と回復演出を行わない

### 要件2: 最大HPによるクランプ

**ユーザーストーリー：** プレイヤーとして、回復してもプレイヤーの最大 HP（`maxHp`）を超えないようにしてほしい。なぜなら最大値を超える HP は仕様外の状態で、HP バーの描画・数値表示・敗北判定の整合性が崩れる原因になるから（敵 HP の 0 クランプと対称）。

#### 受け入れ基準

1. IF 回復計算結果としてプレイヤー HP が `maxHp` を上回る WHEN 回復が適用される THEN the system SHALL プレイヤー HP を `maxHp` にクランプし、`maxHp` を超える値にしない
2. WHEN プレイヤー HP が既に `maxHp` の状態で heal カードのスロットを通過する THEN the system SHALL HP は変化させない（既に上限のため）
3. IF プレイヤー HP が `maxHp` の状態で heal カードのスロットを通過する WHEN ハイライトフェーズに入る THEN the system SHALL 演出（緑フラッシュ・回復数字フロート）は通常通り再生する（演出をスキップしない）

> 補足：要件 2-3 で「HP 変化なしでも演出は出す」と定めているのは、ユーザーが「heal カードを通ったが HP が満タンだったから増えなかった」という結果を視覚的に追えるようにするため。プログラミング学習として「実行されたが効果がなかった」状態を黙示しないことが重要。

### 要件3: プレイヤー側の回復フラッシュ演出

**ユーザーストーリー：** プレイヤーとして、回復が起きた瞬間に「ヒールが効いた」感を視覚的に感じたい。なぜなら HP バーの増加だけでは「いつどのカードで回復したか」を時系列で追いづらいから（攻撃側の白フラッシュ・被弾側の赤フラッシュと色で区別したい）。

#### 受け入れ基準

1. WHEN heal カードのスロットがハイライトフェーズに入り回復処理が走る THEN the system SHALL プレイヤー HP バーまたはその周辺に緑系のフラッシュ演出を起動する
2. WHEN 緑フラッシュ演出が走る THEN the system SHALL 0.2 〜 0.4 秒の短時間で 1 回明滅して自然に終わる（既存の被弾フラッシュと同等のタイムスケール）
3. WHILE 緑フラッシュ中 the system SHALL プレイヤー HP バーの位置・サイズ・既存レイアウトを変更しない（演出は重ね描き）
4. WHEN 演出色が選択される THEN the system SHALL 緑系（敵側のフラッシュが白系、被弾フラッシュが赤系であることに対し回復＝緑で識別できる）にする

### 要件4: 回復量のフロート表示

**ユーザーストーリー：** プレイヤーとして、回復量を数値ですぐに確認したい。なぜなら「+12 回復した」「+5 回復した」とその場で見えると、置いたカードの `power` と結果の対応が直感的にわかり、デバッグ思考に寄与するから（ダメージ数字フロート表示と対称）。

#### 受け入れ基準

1. WHEN heal カードのスロットがハイライトフェーズに入り回復処理が走る THEN the system SHALL プレイヤー HP バー周辺に「+<amount>」形式（例：「+12」）の数字を出現させる
2. WHEN 回復数字が表示される THEN the system SHALL 上方向にゆっくり浮き上がりながら徐々に透明になり、0.6 〜 1.0 秒程度で完全に消える（被弾ダメージフロートと同等の挙動）
3. WHEN 回復数字の文字色が決まる THEN the system SHALL 緑系（被弾の赤系と区別できる色）にする
4. IF プレイヤー HP が `maxHp` の状態で heal カードのスロットを通過する WHEN ハイライトフェーズに入る THEN the system SHALL 実際の HP 変化量（0）ではなく、カードの `power` 値をそのまま「+<power>」として表示する（要件 2-3 と整合）
5. IF 同じ実行シーケンス内で複数の heal カードを通過する場合 WHEN 各回復が順次適用される THEN the system SHALL 各回復数字を独立した要素として描画し、重なって見えてもアニメーションが互いに干渉しない

### 要件5: プレイヤーHPバーの増加アニメーション

**ユーザーストーリー：** プレイヤーとして、HP バーの増え方が一瞬でカクッと変わるより、少しでも滑らかに変化してほしい。なぜなら短時間でも「回復した感」が視覚で伝わると、ヒールの手応えが増してフィードバックがリッチになるから（既存の HP バー減少トランジションと対称）。

#### 受け入れ基準

1. WHEN プレイヤー HP が増加する THEN the system SHALL HP バーの幅変化を 0.2 〜 0.3 秒のトランジションで滑らかに変化させる（既存の `HpBar` 減少トランジション設定を双方向で再利用する）
2. WHEN HP の数値ラベルも変化する THEN the system SHALL バーの幅変化と同時に数値が更新される（バーが伸びるのと同時に数字も増える）

### 要件6: 既存機能との整合性

**ユーザーストーリー：** プレイヤーとして、ヒールカード処理が加わっても既存のカード配置・実行・リセット・拡大／縮小・既存の attack / monster カード処理の挙動が変わらないでほしい。

#### 受け入れ基準

1. WHEN ユーザーがリセットボタンを押す THEN the system SHALL 既存仕様どおり手札とスロット割当を初期化する（プレイヤー HP には触れない、`monster-attack` 要件 5-4 を維持）
2. WHEN ユーザーが拡大トグルを押す、カードをドラッグする等の通常操作を行う THEN the system SHALL プレイヤー HP を変更しない
3. WHEN 実行が完了して通常状態に戻る THEN the system SHALL カードの配置（`slotAssignments`）と手札（`handCards`）を実行前の状態と同一に保つ（既存の play-button 仕様 7-2 を維持）
4. WHEN フローチャートの拡大／縮小切替が走る THEN the system SHALL プレイヤー HP バーの数値表示・回復フラッシュ演出・回復数字も既存のレイアウトを崩さず動作する
5. WHEN 既存の `attack` カード処理および `monster` カード処理が走る THEN the system SHALL ヒールカード機能の追加による副作用を受けず、`attack-processing` および `monster-attack` スペックで定義された挙動を維持する
6. WHEN 実行シーケンス開始時にプレイヤー HP が `maxHp` へリセットされる THEN the system SHALL ヒールカード処理は既存リセット仕様（`monster-attack` 要件 5）と整合し、リセット後の `maxHp` 状態から回復処理が適用される
