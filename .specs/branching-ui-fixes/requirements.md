# 要件定義: 条件分岐ノード周りの UI 修正（branching-ui-fixes）

## はじめに

`map-2-stage-1` で導入した条件分岐ノード（菱形）の周辺で、以下 4 つの表示崩れが報告されている。本機能ではこれらをまとめて修正し、ステージ 2-1 を「設計通りに見える」状態に整える。

1. 菱形ノードに枠線がない（StartNode と統一感がない）
2. 菱形の横頂点（True ハンドル）と隣接スロットの中点で y 座標がずれており、エッジが斜めに傾く
3. 条件分岐後のエッジが斜めに伸び、間にあるロックノードの裏を通過してしまう
4. 菱形内のテキスト（`playerHp > 50`）が「5」と「0」の間で改行され可読性が低い

これらは戦闘体験そのものには影響しないが、プレイヤーが「これは何のノード?」「エッジはどこへ繋がる?」と直感的に理解する妨げになる。視覚的な明確さを優先して修正する。

## 用語

- **菱形ノード**: 条件分岐ノード。`ConditionNode.jsx` で描画される。現在のサイズ 140×140px、`clip-path: polygon(...)` で菱形に切り抜き。
- **横頂点**: 菱形の左右の頂点。React Flow の `Position.Left`（target）と `Position.Right`（True source ハンドル）が配置される。
- **直角折れ線**: 水平/垂直のセグメントだけで構成される折れ線エッジ。React Flow の `step` または `smoothstep` エッジタイプに相当する。

## 要件

### 要件1: 菱形ノードに白い枠線を追加する

**ユーザーストーリー：** プレイヤーとして、条件分岐ノードがフローチャート上の他のマーカー（Start / Goal）と同じデザイン体系で描かれていてほしい。なぜなら、統一感のない見た目はゲームの完成度を損なうから。

#### 受け入れ基準

1. WHEN 条件分岐ノードが描画される THEN the system SHALL `StartNode` と同じ太さ・色の白枠線（`border: 3px solid #f5f5f5` 相当）で菱形を囲む
2. WHEN 菱形ノードが描画される THEN the system SHALL `clip-path` を使った菱形形状で border が削れる問題を回避し、菱形のすべての辺に枠線が表示される
3. WHEN `.active`（実行中点滅）/ `.traversed`（通過済み固定光）の効果が適用される THEN the system SHALL 枠線が消えずに発光と共存する

### 要件2: 菱形の横頂点と隣接スロットの y 座標整合

**ユーザーストーリー：** プレイヤーとして、菱形ノードと隣接ノードを結ぶエッジが水平にまっすぐ伸びていてほしい。なぜなら、斜めのエッジは「分岐ではないのに分岐に見える」という誤解を与えるから。

#### 受け入れ基準

1. WHEN 菱形ノードと通常スロットノード（`SlotNode`）が同じ y 座標で並ぶ THEN the system SHALL 両者の左右ハンドル（target / source）が同じ画面 y 位置に来る
2. WHEN 通常スロットから菱形の左頂点（target）へエッジが伸びる THEN the system SHALL エッジが水平直線で描画される
3. WHEN 菱形の右頂点（True source）から次の通常スロットへエッジが伸びる THEN the system SHALL エッジが水平直線で描画される

実装上の指針：菱形ノードのコンテナサイズと SlotNode のコンテナサイズで「中心 y がずれる」場合は、CSS で位置補正するか、Handle の position オプションで明示的にハンドル座標を揃える。

### 要件3: 条件分岐後のエッジを直角折れ線（L字）スタイルにする

**ユーザーストーリー：** プレイヤーとして、菱形ノードから出る分岐エッジが斜めに伸びずに「下に進んで → 右へ進んで → 合流先へ上から入る」の 3 セグメントの L 字型で描かれてほしい。なぜなら、斜めのエッジが間のノードの裏側を通過して「どこへ繋がるのか」が読み取りにくいから。

#### 受け入れ基準

1. WHEN 条件分岐ノードの **下頂点（False ハンドル）から出るエッジ** が次のノードへ向かう THEN the system SHALL 以下の経路でエッジを描画する：
   - 菱形の下頂点から **垂直下方向** にしばらく伸びる
   - **右方向（進行方向）に直角に曲がる**
   - 合流先ノードの少し手前で **垂直上方向に直角に曲がる**
   - 合流先ノードの上端（top ハンドル）に **垂直に入る**
2. WHEN False 経路にスロットが存在しない（直接合流する）THEN the system SHALL 上記 3 セグメントの L 字経路でそのまま合流先まで伸ばす（中継ノードがなくても直角折れ線を維持）
3. WHEN False 経路にスロットが存在する THEN the system SHALL 菱形 → スロット間、スロット → 次のノード間も直角折れ線で描画する
4. WHEN 条件分岐ノードの **右頂点（True ハンドル）から出るエッジ** が次のノードへ向かう THEN the system SHALL 既存の水平直線エッジで描画する（要件 2 で y が揃っているため、直線になる）
5. WHEN マップ 1 の線形ステージのエッジ THEN the system SHALL 既存の見た目を維持する（直線 or 既存のスタイル、本要件の影響を受けない）

実装上の指針：React Flow には `step` / `smoothstep` の組み込みエッジタイプがあり、これらは直角折れ線を描画する。既存の `AnimatedProgressEdge` カスタムを拡張して、エッジに応じてスタイルを切り替えるか、新規エッジタイプを別途定義する。**合流先ノードに `Position.Top` ハンドル**（`targetHandle: 'top'`）が必要になる場合、`stages.json`（または `stagesLoader.js` の自動生成）で False 経路の合流エッジに `targetHandle: 'top'` を指定する。

### 要件4: 菱形内テキストの数字途中改行を防ぐ

**ユーザーストーリー：** プレイヤーとして、条件式テキスト（例: `playerHp > 50`）が「`playerHp > 5\n0`」のように **数字の途中で改行されない** ようにしてほしい。なぜなら、可読性が著しく落ち、条件式の意味が分かりにくくなるから。

#### 受け入れ基準

1. WHEN 菱形内に条件式テキストが描画される THEN the system SHALL 数字リテラル（連続する数字列）の途中で改行されない
2. WHEN テキスト全体が菱形の表示領域に収まらない THEN the system SHALL **トークン境界**（変数名と演算子の間、演算子と数値の間、空白の位置）で改行する
3. WHEN テキストが極端に長い（変数名が 20 文字以上等）THEN the system SHALL 表示領域からはみ出すよりは、トークン途中で折り返してもよい（極端なケースの対処）

実装上の指針：`word-break: break-all` を使っているため数字の途中でも改行されている。`word-break: normal` + `overflow-wrap: anywhere` または `word-break: keep-all` + `overflow-wrap: break-word` の組み合わせで「英数字シーケンスは1単語として扱う」挙動に変える。

### 要件5: 既存システムとの整合

**ユーザーストーリー：** 開発者として、本修正がマップ 1 のステージ（1-1〜1-4）に影響しないことを保証したい。

#### 受け入れ基準

1. WHEN マップ 1 のステージ（1-1〜1-4、条件分岐ノードを含まない）を表示する THEN the system SHALL エッジが従来通りの見た目で描画される（要件 3 の直角折れ線は条件分岐ノード周辺のみ）
2. WHEN マップ 1 のステージを実行する THEN the system SHALL 通過軌跡（白いネオン光）、アクティブハイライト等の演出が従来通り動作する
3. WHEN 菱形ノードのデザインを変更する THEN the system SHALL `StartNode` / `GoalNode` / `SlotNode` のデザインには影響しない（変更スコープは `ConditionNode.module.css` 内に限定）
