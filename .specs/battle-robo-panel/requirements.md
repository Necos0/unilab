# 要件定義: 戦闘画面のフロチャロボパネル化

## はじめに

戦闘画面の **フローチャートエリア + 手札エリア**（画面中段〜下段）を **フロチャロボの本体（紫色の長方形）に見立てた枠** で囲み、プレイヤーが「フロチャロボのモニター画面上でフローチャートを組み立て、フロチャロボに搭載されているカードを使って戦っている」というメタ的なゲーム世界観を視覚で伝える。フロチャロボはチュートリアル担当のマスコットキャラ（`public/sprites/robo/robo.png`）として既に登場しており、本機能はそのキャラクター性を戦闘画面の UI 構造に拡張する。

利用者は 2 種類いる：
- **プレイヤー（小学生想定）**：チュートリアルで案内してくれたフロチャロボが、戦闘画面でも自分の操作対象として常にそこにいる感じを得て、世界観に没入できる。
- **ゲームデザイナー（プロジェクト作成者）**：フロチャロボの存在感を強める＝チュートリアル以外のシーンでもキャラクターを活用できる仕組みになり、世界観を統一的に運用できる。

### スコープと前提（今回の方針）

- **対象は `.flowchartArea` + `.playerArea` のみ**。`.enemyArea`（画面上段、敵スプライト・HP バー）は完全無変更。
- **装飾はシンプル一択**。タイトルバー風コントロール（最小化・拡大・閉じるボタン）、アンテナ、雷マークなどフロチャロボ固有の追加要素は **今回スコープ外**。紫色の長方形の枠だけを描く。
- **拡大／縮小モードに完全追従**。現状の `flowchart-zoom` 仕様（縮小時 flowchart=35% / 拡大時 80%）を維持したまま、紫枠は内側の `.flowchartArea` + `.playerArea` のサイズ変化に **追従して一緒に拡大／縮小** する。枠だけ固定、枠だけ非表示などの分岐は実装しない。
- **既存の機能を一切壊さない**。React Flow のフローチャート描画・拡大・dnd-kit のドラッグ&ドロップ・手札のレンダリング・HP バー・ボタン群（Play / Reset / ZoomButton / BackToMap）等はすべて従来どおり動作する。
- **色は `robo.png` の本体紫を基調にする**。具体値はピクセルから採取して `--robo-frame-color` 等の CSS 変数として定義し、設計時に確定する。

## 要件

### 要件1: フロチャロボパネルの基本構造
**ユーザーストーリー：** プレイヤーとして、戦闘画面を開いたときに「フローチャートと手札がフロチャロボの紫色の本体（モニターパネル）に収まっている」と一目で分かる枠が表示されてほしい。なぜなら、自分が操作しているのはフロチャロボのパネルである、という世界観をすぐに感じ取れるから。

#### 受け入れ基準
1. WHEN 戦闘画面が表示される THEN the system SHALL `.flowchartArea` と `.playerArea` を **1 つの紫色の枠（フロチャロボパネル）の中** に内包する形でレイアウトする
2. WHEN フロチャロボパネルを描画する THEN the system SHALL フロチャロボ画像（`robo.png`）の本体紫色と視覚的に同じ色相・明度のフレームを使用する
3. WHEN フロチャロボパネルを描画する THEN the system SHALL `.enemyArea`（画面上段）には一切手を加えない（敵スプライト・HP バー・各種オーバーレイすべて従来どおり）
4. IF プレイヤーが戦闘画面以外（マップ画面・タイトル画面等）にいる場合 THEN the system SHALL NOT フロチャロボパネルを描画しない（本機能は戦闘画面専用）

### 要件2: 装飾レベルの制約（シンプルな枠のみ）
**ユーザーストーリー：** ゲームデザイナーとして、紫枠だけのシンプルなデザインに留めたい。なぜなら、タイトルバー風コントロールやアンテナ、雷マーク等の装飾を入れると視覚情報が増えてフローチャート本体への注目度が下がるし、実装コストも上がるから。

#### 受け入れ基準
1. WHEN フロチャロボパネルを描画する THEN the system SHALL **長方形の紫色の枠**（border / box-shadow / 背景色のいずれか／組み合わせ）以外の装飾を **描画しない**
2. IF タイトルバー風のウィンドウコントロール（最小化・拡大・閉じる）が必要になった場合 THEN the system SHALL NOT 本機能では実装せず、別機能として切り出す
3. IF アンテナ・雷マークなどフロチャロボ固有の装飾が必要になった場合 THEN the system SHALL NOT 本機能では実装せず、将来拡張として保留する
4. WHEN フロチャロボパネルの内側に既存 UI（フローチャート・手札・HP バー・ボタン群）が表示される THEN the system SHALL 既存 UI のレイアウト・サイズ・配置を本機能のために変更しない

### 要件3: 拡大／縮小モードへの追従
**ユーザーストーリー：** プレイヤーとして、フローチャートを拡大ボタンで拡大したときも、紫の枠が一緒に拡大して「フロチャロボの画面が大きくなった」かのように振る舞ってほしい。なぜなら、拡大時に枠だけ取り残されたり消えたりすると、世界観の没入感が途切れるから。

#### 受け入れ基準
1. WHEN プレイヤーが `ZoomButton` で縮小状態から拡大状態へ切り替える THEN the system SHALL `.flowchartArea` の `flex-grow: 35 → 80` のトランジション（既存の 250ms）に追従して、フロチャロボパネルの紫枠も **一緒に拡大** する
2. WHEN プレイヤーが拡大状態から縮小状態へ切り替える THEN the system SHALL 同様にトランジションに追従して紫枠も **一緒に縮小** する
3. WHILE 拡大／縮小のトランジション中 the system SHALL 紫枠の連続性を保つ（途中で消えたり、ガクッと位置が飛んだりしない）
4. IF 拡大状態で `.enemyArea` の高さが 0 に潰れる場合 THEN the system SHALL フロチャロボパネルの上端が `.enemyArea` のあった位置まで自然に持ち上がる（敵エリアの透明領域 / 罫線等の処理は要件 1-3 で `.enemyArea` 無変更としているため自動的に成立する）

### 要件4: 既存ステージへの非破壊性
**ユーザーストーリー：** 開発者として、既存ステージ（1-X / 2-X / 3-X / 4-X / 5-X 等）の戦闘画面が、フローチャート種別やステージ規模に関係なく本機能導入前と同じように動作してほしい。なぜなら、デグレ検証コストが上がるから。

#### 受け入れ基準
1. WHEN 任意の既存ステージで戦闘画面を開く THEN the system SHALL フローチャートのレイアウト（slot 配置・cond ノード・merge ノード・エッジ）を本機能導入前と完全に同一に描画する
2. WHEN 任意の既存ステージでカードをドラッグ&ドロップする THEN the system SHALL dnd-kit のヒットテスト・ドロップターゲット・スロットへの吸着挙動を一切変更しない
3. WHEN 任意の既存ステージで実行ボタン（Play）を押す THEN the system SHALL `battleStore.startExecution` の動的エッジ追跡・実行アニメーション・通過演出を一切変更しない
4. WHEN フローチャートを拡大／縮小する THEN the system SHALL `FlowchartArea` の `ResizeObserver` ＋ `fitBounds` / `setViewport` 計算を一切変更しない（追従はあくまで CSS レベルの flex / size 変化として行う）

### 要件5: 色とスタイルの統一基盤
**ユーザーストーリー：** 開発者として、フロチャロボパネルの紫色を 1 箇所で定義したい。なぜなら、将来「フロチャロボの色味調整」「テーマ色変更」「他画面（タイトル・マップ等）にも同じ紫を使う」のような変更が来たときに 1 箇所変えれば全体に反映できるから。

#### 受け入れ基準
1. WHEN フロチャロボパネルの色を定義する THEN the system SHALL CSS カスタムプロパティ（`--robo-frame-color` 等）として **1 箇所** に書く
2. WHEN フロチャロボパネルの紫色を `robo.png` から採取する THEN the system SHALL 本体メイン部分の代表色（暗めの紫 1 色、または外枠ダーク＋内側ライトの 2 階調）をピクセルから取得する
3. WHEN CSS カスタムプロパティを定義する場所を決める THEN the system SHALL 既存の `frontend/src/index.css`（グローバル CSS）または BattleScreen 内の Module CSS に定義する（実装フェーズで適切な場所を確定）
4. IF 将来「フロチャロボの色を変更したい」「テーマ色対応」が来る場合 THEN the system SHALL CSS カスタムプロパティの値を変更するだけで全パネルに反映できる構造を維持する

### 要件6: アクセシビリティと既存演出への配慮
**ユーザーストーリー：** プレイヤーとして、紫枠が追加されても、既存の戦闘演出（ダメージフロート・勝利オーバーレイ・失敗オーバーレイ・カード光彩・実行中のスロット発光等）が見えづらくなったり、紫枠で隠れたりしないでほしい。

#### 受け入れ基準
1. WHEN プレイヤーがカードをドラッグして手札スロットの外でドロップする THEN the system SHALL カードがフロチャロボパネルの紫枠の上をスムーズに通過して、視覚的に隠れない
2. WHEN 勝利演出（`VictoryClearOverlay`）または失敗演出（`BattleFailOverlay`）が表示される THEN the system SHALL オーバーレイがフロチャロボパネルより前面に描画され、紫枠で隠れない
3. WHEN フローチャート実行中にスロットの `.active` 発光・カードの `.flashing` 演出が走る THEN the system SHALL 紫枠の box-shadow / outline 等と視覚的に競合しない（演出が見えづらくなる場合は本機能側の box-shadow を控えめにする）
4. WHILE プレイヤーが手札パネルで HP バー・ガード残量バー等のステータスを確認する the system SHALL 紫枠がそれらの上に被さって読みづらくならない

### 要件7: モバイル／タブレット等の小画面への配慮
**ユーザーストーリー：** プレイヤーとして、小さい画面（iPad・スマートフォン等）でも紫枠が破綻せずに表示されてほしい。

#### 受け入れ基準
1. WHEN 画面幅が狭い端末（おおむね `1024px` 以下）で戦闘画面を表示する THEN the system SHALL 紫枠の幅・余白を画面サイズに対して破綻しないよう（横スクロールが出ない、内側の UI を圧迫しない範囲で）設定する
2. WHEN ブラウザの zoom（拡大）機能で UI が拡大される THEN the system SHALL 紫枠の太さや角丸も `em` / `rem` 等の相対単位で追従する（または最低限固定 px でも問題にならない範囲に収める）
3. IF 既存の戦闘画面の最小サイズ・最小フォントサイズ等の制約がある場合 THEN the system SHALL それらを本機能のために変更しない

---

## 対象外（今回のスコープ外）

- **タイトルバー風コントロール**（最小化・拡大・閉じるボタンの表示）。本機能では描画しない。
- **アンテナ・雷マーク等のフロチャロボ固有装飾**。本機能では描画しない。
- **タイトル画面・マップ画面・チュートリアル画面等への紫枠展開**。本機能は戦闘画面専用。
- **フロチャロボ自体の表情・アニメーション**（瞬き・喜怒哀楽の表情切替等）。本機能では扱わない。
- **音響演出**（パネルからの効果音等）。本機能では扱わない。
- **テーマ色変更機能**（プレイヤーが紫以外の色を選べる等）。CSS カスタムプロパティ化までは要件 5 でカバーするが、UI からの変更機能は対象外。
- **横長画面・縦長画面など特殊レイアウト対応**。要件 7 の範囲（小画面で破綻しない）に留め、特殊レイアウト用の追加デザインはしない。

## 用語

| 用語 | 意味 |
|---|---|
| フロチャロボ | チュートリアル担当のマスコットキャラ。本体は紫色の CRT モニター風で、画面に水色の顔が表示される。画像は `frontend/public/sprites/robo/robo.png`。 |
| フロチャロボパネル | 本機能で追加する、`.flowchartArea` + `.playerArea` を囲む紫色の枠。フロチャロボの本体（モニター部分）を模す。 |
| `--robo-frame-color` | 紫枠の色を定義する CSS カスタムプロパティの仮称（実装フェーズで確定）。 |
| 拡大／縮小モード | 既存の `flowchart-zoom` 仕様。`ZoomButton` で `battleStore.isExpanded` を切り替えると `.flowchartArea` の `flex-grow` が 35→80（拡大）または 80→35（縮小）に推移する。 |
