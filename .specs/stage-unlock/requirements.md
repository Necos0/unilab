# 要件定義: ステージ解放機能

## はじめに

現状はマップ上の全ランドマーク（戦闘可能な全ステージ）に自由にアクセスでき、進行感が無い。本機能では、ステージを直列に解放していく仕組みを導入する。初期は各ワールドの `*-1` ステージのみ挑戦可能とし、ステージをクリアするごとに次のステージ（`*-2` → `*-3` → `*-4` …）が解放される。未解放のステージはランドマーク名の上に「鎖＋南京錠」のロックアイコンを重ねて表示し、クリックを受け付けない。クリア済みステージは詳細パネル（`LandmarkDetail`）に「クリア済み」のラベルを表示し、再挑戦は可能なままとする。

解放／クリアの進行はゲームセッション中のメモリ（zustand）で管理し、ブラウザリロードでリセットする（永続化は本スペックの対象外）。

## 要件

### 要件1: 初期状態の制限

**ユーザーストーリー：** プレイヤーとして、ゲーム開始直後は最初のステージ（`*-1`）にだけ挑戦できるようにしたい。なぜなら、いきなり全ステージに挑めると進行の達成感が薄れ、難易度設計も崩れるから。

#### 受け入れ基準
1. WHEN マップ画面を初回表示した THEN the system SHALL `stageId` が `*-1` 形式（番号部分が `1`）のランドマークのみを「解放済み」として扱う。
2. WHEN マップ画面を初回表示した THEN the system SHALL `stageId` が `*-2` 以降のランドマークを「未解放」として扱い、それぞれのランドマーク名（`LandmarkScroll`）の上に鎖＋南京錠のロックアイコンを重ねて表示する。
3. WHERE ランドマークが `stageId` を持たない（`village_gate` 等の純粋な経由地）場合、the system SHALL ロック表示も解放判定も行わず常に通常表示とする（移動先としては従来どおり機能する）。

### 要件2: 未解放ステージのクリック抑止

**ユーザーストーリー：** プレイヤーとして、未解放のステージは選んでも反応しないことが見た目で分かるようにしたい。なぜなら、クリックできるかどうかが視覚的に区別できないと、操作がもどかしく感じるから。

#### 受け入れ基準
1. WHILE ランドマークが「未解放」状態である the system SHALL そのランドマークのクリックを移動要求（`requestMove`）として発火しない。
2. WHILE ランドマークが「未解放」状態である the system SHALL そのランドマークの「たたかう」ボタンを表示しない（到着できないため `LandmarkDetail` 自体が表示されない経路で十分だが、万一到着判定が真になっても戦闘開始させない）。
3. WHEN ロックアイコンが表示されている THEN the system SHALL そのアイコンのクリックも何の動作も発火しない（移動要求にもならない）。

### 要件3: クリア記録

**ユーザーストーリー：** ゲーム進行管理として、どのステージを既にクリアしたかを記録したい。なぜなら、クリア済みかどうかで「次のステージ解放」「クリア済みラベル表示」など複数のUI挙動が変わるから。

#### 受け入れ基準
1. WHEN プレイヤーが `VictoryClearOverlay`（CLEAR! 演出）の「マップへ戻る」ボタンを押した THEN the system SHALL そのバトルの `stageId` を「クリア済み」として記録する。
2. IF ある `stageId` が既に「クリア済み」として記録済み WHEN プレイヤーが同ステージを再挑戦して勝利し「マップへ戻る」を押した THEN the system SHALL 重複した記録は追加せず、状態を変化させない（ただし要件5の演出は再生しない）。
3. WHEN 戦闘画面右上のテスト用「マップへ戻る」ボタン（`BackToMapButton`）を押した THEN the system SHALL クリア記録を追加しない（勝利演出を経由していないため、クリアとして扱わない）。
4. WHEN プレイヤーが敗北した（プレイヤーHP=0で戦闘終了）または引き分けで戦闘を終えた THEN the system SHALL クリア記録を追加しない（本機能の対象外。VictoryClearOverlay の経路のみがクリアトリガー）。

### 要件4: 次ステージの直列解放

**ユーザーストーリー：** プレイヤーとして、ステージをひとつクリアするたびに次のステージが解放されてほしい。なぜなら、進行に応じて挑戦できる場所が増えていく実感がほしいから。

#### 受け入れ基準
1. WHEN ある `stageId`（例：`1-2`）が新たに「クリア済み」として記録された THEN the system SHALL 同ワールド内で番号が 1 大きい `stageId`（例：`1-3`）を「解放済み」状態へ遷移させる。
2. IF 番号が 1 大きいステージが `stages.json` に存在しない（例：`1-4` クリア時の `1-5`）WHEN クリアが記録された THEN the system SHALL 解放処理は何もしない（最終ステージとして扱う）。
3. IF 番号が 1 大きいステージが既に「解放済み」状態である（リプレイ時など）WHEN クリアが記録された THEN the system SHALL 解放処理は何もしない（要件5の演出も再生しない）。
4. WHEN 解放処理が走った THEN the system SHALL `*-1` ステージは初期状態として常に解放済みであるため、新規解放の対象から外す（直列の起点として永続的に解放）。

### 要件5: 解放アニメーション

**ユーザーストーリー：** プレイヤーとして、ステージが解放された瞬間が視覚的に分かりやすく演出されてほしい。なぜなら、達成のフィードバックがあると次の挑戦への動機づけになるから。

#### 受け入れ基準
1. WHEN 戦闘画面から「マップへ戻る」が押されてマップ画面が再表示され、新たに解放されたステージがある THEN the system SHALL マップ画面の表示直後に、当該ランドマークのロックアイコンをフェードアウトさせるアニメーションを再生する。
2. WHEN フェードアウトアニメーションが完了した THEN the system SHALL 当該ランドマークを通常の解放済み表示（ロックなし）に切り替え、クリック可能とする。
3. WHILE 解放アニメーション再生中である the system SHALL マップ画面上の全ランドマーククリックを無効化し、移動要求を発火しない。
4. WHILE 解放アニメーション再生中である the system SHALL プレイヤーの自動移動・「たたかう」ボタンも無効化する（クリック不能扱いと整合させる）。
5. IF クリアによって新たに解放されたステージが無い（最終ステージのクリア・既に解放済みステージのリプレイ）WHEN マップ画面が再表示された THEN the system SHALL 解放アニメーションを再生せず、クリック抑止も発動しない。

### 要件6: クリア済みラベル表示

**ユーザーストーリー：** プレイヤーとして、どのステージを既にクリアしたかをマップ上で確認したい。なぜなら、進捗を一目で把握できると、次にどのステージへ向かうか判断しやすいから。

#### 受け入れ基準
1. WHILE あるステージが「クリア済み」状態である the system SHALL 当該ランドマークの `LandmarkDetail` パネルの一番上（既存の「難易度」ラベルより上）に「クリア済み」というテキストを表示する。
2. WHILE あるステージが「クリア済み」状態である the system SHALL 「たたかう」ボタンを引き続き有効に保ち、再挑戦を許可する。
3. IF ステージが未クリアまたは未解放 THEN the system SHALL 「クリア済み」ラベルを表示しない。
4. WHILE ステージが未解放（ロック中）である the system SHALL `LandmarkDetail` パネル自体を到着判定に応じて表示するが、要件2-2 によって「たたかう」ボタンは押せない（パネルの存在自体は維持してもしなくても良いが、未解放ランドマークは到着不能であるため実質非表示となる）。

### 要件7: 状態の保持範囲

**ユーザーストーリー：** 開発初期段階のため、進行状態を実装最小限で保持したい。なぜなら、永続化（localStorage）まで含めるとスコープが広がるため、まずはセッション中の動作を完成させたいから。

#### 受け入れ基準
1. WHEN ブラウザのページを再読み込みした THEN the system SHALL クリア記録・解放状態を初期状態（`*-1` のみ解放）にリセットする。
2. WHEN マップ画面 ↔ 戦闘画面を行き来した THEN the system SHALL クリア記録・解放状態を保持する（zustandストアはアンマウントされない）。
3. WHERE 将来 localStorage / バックエンド永続化を導入する場合、the system SHALL ストアの公開APIを変更せずに永続層を差し替えられる構造とする（本要件は将来拡張への配慮であり、本スペックでは実装しない）。
