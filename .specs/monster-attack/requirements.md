# 要件定義: モンスターカードによるプレイヤー被弾処理

## はじめに

ステージのフローチャート上に **モンスターカード** という新しいカード種別を導入し、プレイヤーが実行中にそのスロットを通過すると、そのステージのモンスター（既存の敵：slime / wolf / knight / golem）から `power` 値分のダメージを受ける、という戦闘ロジックを追加する。

モンスターカードは `attack` / `guard` / `heal` と同列のカード種別だが、**ステージ定義側で特定スロットに固定配置される** 点が異なる。ユーザーは手札からそのスロットへドロップできず、配置済みのモンスターカードをドラッグで外すこともできない。デザイン班から本デザインを受け取り次第 `public/cards/monster.png` を差し替える前提で、本スペックの実装範囲では仮のデモビジュアルを用意する。

本スペックの対象は「モンスター由来の被弾ダメージをプレイヤー HP から減算する」までのロジックと演出。プレイヤー HP が 0 になった際の **敗北判定・敗北演出・実行中断** は別スペックで扱う。デモは `1-2`（ウルフ）ステージにモンスターカードを 1 枚配置することで動作確認する。対象利用者はステージをプレイするプレイヤー本人。

## 要件

### 要件1: モンスターカードのカード種別

**ユーザーストーリー：** プレイヤーとして、フローチャート上に「これは敵からの攻撃を受ける枠だ」と一目でわかるカードがあってほしい。なぜなら attack / guard / heal のような自分が選ぶカードとは性質が違う「外から課される試練」を視覚的に区別できることで、フローチャートを読み解く手がかりになるから。

#### 受け入れ基準

1. WHEN ステージ定義でカード種別 `monster` のカードが参照される THEN the system SHALL 既存の attack / guard / heal と同じく `id`（`'monster'`）と `power`（被ダメージ量）の組で表現できる
2. WHEN モンスターカードが画面に描画される THEN the system SHALL 他のカード種別と区別できる仮のデモビジュアル（後でデザイン差し替え可能な形式）で表示する
3. IF `public/cards/monster.png` が後から正規アセットに差し替えられる WHEN 戦闘画面が再描画される THEN the system SHALL コード変更なしで新ビジュアルが反映される（既存カードと同じ `/cards/<id>.png` 参照規約に従う）

### 要件2: ステージ定義によるモンスターカードの固定配置

**ユーザーストーリー：** ステージ作成者として、特定のステージ・特定のスロットに「ここを通ると敵から攻撃を受ける」配置をあらかじめ決めておきたい。なぜならモンスターカードはステージ設計の一部であり、難易度や経路選択の意味付けに直結するから。

#### 受け入れ基準

1. WHEN `stages.json` のステージ定義に「特定スロットへモンスターカードを固定配置する」記述がある THEN the system SHALL 戦闘画面マウント時にそのスロットへモンスターカードが置かれた状態で初期化する
2. WHEN ユーザーが手札のカードを **モンスターカードが固定配置されたスロット** へドラッグドロップしようとする THEN the system SHALL そのドロップを拒否する（割り当て更新を行わない）
3. WHEN ユーザーが **モンスターカードが固定配置されたスロット** からカードをドラッグしようとする THEN the system SHALL そのドラッグを拒否する（モンスターカードはスロットから外せない）
4. WHEN ユーザーがリセットボタンを押す THEN the system SHALL モンスターカードの固定配置を維持したまま、ユーザーが配置した他のカードのみ手札へ戻す
5. IF デモステージ `1-2` がロードされる WHEN 戦闘画面がマウントされる THEN the system SHALL `1-2` の所定スロットにモンスターカード（`power` 値はステージ JSON で指定）を配置した状態で表示する

### 要件3: プレイヤー HP の状態管理

**ユーザーストーリー：** プレイヤーとして、被弾の累積結果として現在の自分の HP を把握できるようにしたい。なぜならフローチャートを書き換えてモンスターカードを通る回数を変えたとき、その結果が自分の HP にどう反映されるかを観察することが、デバッグ駆動の学習サイクルの根幹だから。

#### 受け入れ基準

1. WHEN 戦闘画面がマウントされる THEN the system SHALL プレイヤー HP を `player.json` に定義された `maxHp` 値で初期化する
2. WHILE 戦闘画面が表示されている the system SHALL 現在のプレイヤー HP を `battleStore` でグローバルに保持し、HP バー描画と数値表示で同じ値を参照する
3. IF ダメージ計算結果としてプレイヤー HP が 0 を下回る WHEN ダメージが適用される THEN the system SHALL プレイヤー HP を 0 にクランプし、マイナス値にしない
4. IF プレイヤー HP が 0 に到達する WHEN ダメージが適用される THEN the system SHALL HP を 0 で保持するに留め、敗北判定・実行中断・敗北演出は **行わない**（別スペックで扱う）

### 要件4: モンスターカードによるダメージ適用

**ユーザーストーリー：** プレイヤーとして、フローチャート実行中にモンスターカードのスロットを通ったとき、そのカードの `power` 値が自分にダメージとして反映されてほしい。なぜなら通る回数や経路選択がプレイヤー HP の結果に直結することで、「どう経路を組み立てれば被弾を抑えられるか」というプログラミング的因果関係を学べるから。

#### 受け入れ基準

1. WHEN 実行シーケンス進行中にスロットノードがハイライトフェーズに入る AND そのスロットに `id === 'monster'` のカードが配置されている THEN the system SHALL そのカードの `power` 値をプレイヤー HP から減算する
2. IF 配置されているカードが `monster` 以外（`attack` / `guard` / `heal` 等）の場合 WHEN スロットノードがハイライトフェーズに入る THEN the system SHALL プレイヤー HP は変化させない
3. WHEN ダメージが適用される THEN the system SHALL ハイライトフェーズの開始タイミングに合わせて減算を実行し、被弾演出（要件 7）と HP バー減少を時系列で同期させる
4. IF `power` が 0 または欠損している WHEN モンスターカードのスロットがハイライトフェーズに入る THEN the system SHALL ダメージ適用と被弾演出を行わない

### 要件5: 実行ごとのプレイヤー HP リセット

**ユーザーストーリー：** プレイヤーとして、実行ボタンを押すたびにプレイヤー HP が初期値（`maxHp`）からスタートしてほしい。なぜなら本ゲームは「コードを書く→実行→結果を見る→修正→再実行」のループを通じて目的の動作を作り上げるデバッグ駆動の学習体験であり、各実行はフレッシュな状態から走らないと「フローチャートを修正したらどう結果が変わるか」を比較できないから（敵 HP 側と対称の挙動）。

#### 受け入れ基準

1. WHEN ユーザーが実行ボタンを押し、実行シーケンスが開始される THEN the system SHALL プレイヤー HP を `maxHp` に戻してからシーケンスを開始する
2. WHEN 実行シーケンスが完了する THEN the system SHALL プレイヤー HP をシーケンス終了時点の値（0〜`maxHp`）で保持する（次の実行が始まるまで動かさない）
3. WHILE 実行終了後〜次の実行開始までの通常状態 the system SHALL プレイヤー HP の現在値を可視化したまま維持し、ユーザーが「今回の実行でいくつ削られたか」を確認できる
4. IF ユーザーがリセットボタンを押す（カード配置の初期化）WHEN 通常状態 THEN the system SHALL プレイヤー HP には触れない（HP のリセットは次の実行開始時に自動で起きるため、リセットボタンの責務は配置のみ）

### 要件6: プレイヤー HP バーの動的化と数値表示

**ユーザーストーリー：** プレイヤーとして、自分の HP の現在値と最大値を数値で正確に把握したい。なぜなら「あと何ダメージで死ぬか」を逆算しながら経路を組み立てる必要があり、視覚的なバーの長さだけでは正確な計算がしづらいから。

#### 受け入れ基準

1. WHEN プレイヤー HP バーが描画される THEN the system SHALL 現状の静的な `playerMaxHp/playerMaxHp` 表示から、`battleStore` の現在値を購読する動的表示に置き換える
2. WHEN プレイヤー HP バーが描画される THEN the system SHALL `currentHp/maxHp` 形式の数値ラベルを敵 HP バーと同じレイアウト・スタイルで併記する（既存の `.hpText` 相当を流用）
3. WHEN プレイヤー HP が変化する THEN the system SHALL バーと数値ラベルが共に同期して更新される
4. WHEN プレイヤー HP が減少する THEN the system SHALL 既存の `HpBar` の幅トランジション設定（敵側と共通）に従って滑らかに変化する

### 要件7: プレイヤー被弾演出

**ユーザーストーリー：** プレイヤーとして、自分が敵から攻撃を受けた瞬間に「ダメージを食らった」感を視覚的に感じたい。なぜなら HP バーの減少だけでは「いつ・どのモンスターカードで・いくつ削られたか」を時系列で追いづらいから。

#### 受け入れ基準

1. WHEN モンスターカードのスロットがハイライトフェーズに入りダメージが適用される THEN the system SHALL ダメージ量を「-<damage>」形式（例：「-8」）の数字としてプレイヤー HP バー周辺にフロート表示する
2. WHEN ダメージ数字が表示される THEN the system SHALL 上方向にゆっくり浮き上がりながら徐々に透明になり、0.6 〜 1.0 秒程度で完全に消える
3. WHEN モンスターカードのスロットがハイライトフェーズに入りダメージが適用される THEN the system SHALL プレイヤー HP バーまたはその周辺に「被弾した」ことが伝わる短い演出（赤フラッシュ・微小シェイク等）を 0.2 〜 0.4 秒で 1 回再生する
4. IF 同じ実行シーケンス内で複数のモンスターカードを通過する場合 WHEN 各ダメージが順次適用される THEN the system SHALL 各ダメージ数字を独立した要素として描画し、重なって見えてもアニメーションが互いに干渉しない
5. WHEN プレイヤー被弾演出が走る THEN the system SHALL 演出色は赤系（敵側のフラッシュが白系であることと対比）で、プレイヤー側であることが視覚的に区別できるようにする

### 要件8: 既存機能との整合性

**ユーザーストーリー：** プレイヤーとして、モンスターカード処理が加わっても既存のカード配置・実行・リセット・拡大／縮小の挙動が変わらないでほしい。

#### 受け入れ基準

1. WHEN ユーザーがリセットボタンを押す THEN the system SHALL ユーザー配置のカードのみ手札へ戻し、モンスターカードの固定配置とプレイヤー HP には触れない（要件 2-4 / 5-4）
2. WHEN ユーザーが拡大トグルを押す、カードをドラッグする等の通常操作を行う THEN the system SHALL プレイヤー HP を変更しない
3. WHEN 実行が完了して通常状態に戻る THEN the system SHALL モンスターカードの固定配置とユーザー配置カードの状態は実行前と同一に保つ（既存の play-button 仕様 7-2 を維持）
4. WHEN フローチャートの拡大／縮小切替が走る THEN the system SHALL プレイヤー HP バーの数値表示・被弾演出・ダメージ数字も既存のレイアウトを崩さず動作する
5. WHEN 既存の `attack` カード処理が走る（敵へのダメージ適用） THEN the system SHALL モンスターカード機能の追加による副作用を受けず、`attack-processing` スペックで定義された挙動を維持する
