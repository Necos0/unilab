# 要件定義: フローチャートのスタート／ゴールマーカー

## はじめに

フローチャートの両端に **「ここから始まる／ここで終わる」を視覚的に示すマーカー** を追加する。最左の **スタートマーカー（右向き矢印「→」）** と最右の **ゴールマーカー（モノクロ旗）** は、既存のカード配置用スロットと違いプレイヤーがカードを置く対象ではなく、純粋にフローの起点と終点を示す装飾要素。`stages.json` に位置を明示し、既存スロットと矢印（エッジ）で接続する。将来複数行になっても「最上段左にスタート、最下段右にゴール」が常に固定の構造として続く前提。本機能はステージをプレイするプレイヤー本人と、ステージを定義する開発者を対象とする。

## 要件

### 要件1: スタート／ゴールマーカーの表示

**ユーザーストーリー：** プレイヤーとして、フローチャートのどこから処理が始まりどこで終わるのかを視覚的に把握したい。なぜなら配置したカードがどの順番で実行されるかをひと目で理解する必要があるから。

#### 受け入れ基準

1. WHEN 戦闘画面でフローチャートが描画される THEN the system SHALL `stages.json` に定義されたスタートマーカー位置に右向き矢印（「→」）を表示する
2. WHEN 戦闘画面でフローチャートが描画される THEN the system SHALL `stages.json` に定義されたゴールマーカー位置にモノクロの旗マーカー（🏁 風のシンプルな絵柄）を表示する
3. WHILE フローチャートが描画されている the system SHALL スタート／ゴールマーカーを、既存のカード配置スロット（点線の矩形枠）とは異なる外観で表示し、配置先ではないことが視覚的にわかるようにする

### 要件2: `stages.json` での定義

**ユーザーストーリー：** ステージ作成者として、各ステージごとにスタート／ゴールの位置を明示的に指定したい。なぜなら「最上段左がスタート」というルールを暗黙的に計算するより、ステージ定義ファイルに直接書いてあるほうがレビューや編集がしやすいから。

#### 受け入れ基準

1. WHEN `stages.json` の各ステージにスタート／ゴール定義（位置情報を含む）が記載される THEN the system SHALL これを読み取って表示位置に反映する
2. WHEN スタート／ゴールを指す矢印が `stages.json` のエッジ配列に明示される THEN the system SHALL これを既存スロット間のエッジと同じ仕組みで描画する
3. IF ステージにスタートまたはゴール定義が無い場合 WHEN ステージを表示する THEN the system SHALL 開発者警告（`console.warn`）を出しつつ、該当マーカーを描画しない（既存スロット・エッジは通常通り描画する／クラッシュさせない）

### 要件3: 既存スロットとのエッジ接続

**ユーザーストーリー：** プレイヤーとして、スタートからゴールまでが一つの連続した流れとしてつながって見えてほしい。なぜなら断絶した複数のグラフではなく「実行順序のあるパイプライン」として認識したいから。

#### 受け入れ基準

1. WHEN フローチャートが描画される THEN the system SHALL `stages.json` の `edges` 定義に従い、スタートマーカー → 最初のカード配置スロットの矢印、および最後のカード配置スロット → ゴールマーカーの矢印を描画する
2. WHEN ステージ定義のエッジが存在しないスタート／ゴールを参照する THEN the system SHALL 既存の整合性チェック（参照不正なエッジは除外しつつ `console.warn` で警告）と同じ方針で扱う
3. WHEN スタート／ゴールマーカーがエッジで接続される THEN the system SHALL 矢印の見た目（太さ・色・矢じり）を既存スロット間エッジと同じスタイルで描画する

### 要件4: ドロップターゲットからの除外

**ユーザーストーリー：** プレイヤーとして、スタート／ゴールにはカードを置けないようにしてほしい。なぜならこれらは配置先ではなく単なるマーカーで、誤って置こうとした時に「置けるかも」と期待を持たせる挙動はストレスになるから。

#### 受け入れ基準

1. IF カードがスタートマーカーまたはゴールマーカー上でドロップされる WHEN ドロップが確定する THEN the system SHALL 当該マーカーを有効なドロップ先として扱わず、カードはドロップ元（手札またはドラッグ元スロット）に戻す
2. WHILE カードがドラッグされている the system SHALL スタート／ゴールマーカーをドロップ可能なハイライト対象として扱わない（カード配置スロットだけがハイライトされる）
3. IF ユーザーがスタート／ゴールマーカーをドラッグ操作しようとする THEN the system SHALL ドラッグを開始しない（カードのドラッグ対象としても、ノードのドラッグ対象としても無効）

### 要件5: 拡大／縮小ズームとの整合性

**ユーザーストーリー：** プレイヤーとして、フローチャートの拡大／縮小トグルを切り替えた際、スタート／ゴールマーカーも既存スロットと一緒にスケールして欲しい。なぜなら別々にスケールすると不自然な見た目になるから。

#### 受け入れ基準

1. WHEN フローチャートが拡大／縮小切替される THEN the system SHALL スタート／ゴールマーカーも既存スロットと同じ React Flow ビューポート変換（`fitView` のスケール）に従って同時にスケール変化する
2. WHILE フローチャートが縮小状態 the system SHALL スタート／ゴール込みのフローチャート全体が縮小エリアに収まるよう自動フィットする
3. WHILE フローチャートが拡大状態 the system SHALL スタート／ゴールも含めたフローチャート全体に対して `minZoom: 1` 下限の自動フィットを適用する

### 要件6: リセットの影響範囲

**ユーザーストーリー：** プレイヤーとして、リセットボタンを押してもスタート／ゴールマーカーは消えたり位置が変わったりしないでほしい。なぜならこれらはステージ構造の一部であって、配置をやり直すたびに変動するものではないから。

#### 受け入れ基準

1. WHEN ユーザーがリセットボタンを押す THEN the system SHALL スタート／ゴールマーカーの表示状態（位置・見た目）を変更しない
2. WHEN ユーザーがリセットボタンを押す THEN the system SHALL 既存仕様どおりカード配置スロットの中身のみを初期化する（スタート／ゴールにはそもそもカードが入らないので影響は無い）
