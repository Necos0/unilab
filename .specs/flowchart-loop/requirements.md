# 要件定義: フローチャートのループ（while）構文

## はじめに

戦闘画面のフローチャートに **while ループ（繰り返し）構造** を導入する。これまで
`flow` ショートカットは「線形」と「条件分岐（分岐後に auto-merge する非循環ツリー）」
のみを表現できたが、本機能では `flow` に **loop 構文** を新設し、ローダーが
「条件ノードの手前に置く合流ノード（merge）＋ループボディ＋ボディ末尾から merge への
戻りエッジ」を含む完全形式へ自動展開できるようにする。あわせて、条件ノードの
true / false 出口方向を `stages.json` で任意指定できるようにし、ループ特有の
「成立で下＝脱出、不成立で横＝ボディ」のような配置を宣言的に書けるようにする。

利用者は 2 種類いる：
- **ステージデザイナー**（このプロジェクトの作成者）：`stages.json` にループステージを
  少ない記述で追加できるようになる。
- **プレイヤー（小学生想定）**：`stage 4-1` で「条件が満たされるまで同じ処理を繰り返す」
  というプログラミングの基礎概念をパズルとして体験する。

最初の具体的な成果物は **stage 4-1**（`start → merge → cond(敵HPが0?) →
〔false: 右へ「空き → monster:20 → 空き」→ 上側を通って merge へ戻る〕／
〔true: 下へ Goal〕`、手札＝攻撃(15)・防御(10)、敵 HP=60）。

### スコープと前提（今回の方針）

- **対象トポロジは「単一 while ループ」のみ**。入れ子ループ、ループボディ内での
  条件分岐（cond のネスト）は **今回の対象外**（将来拡張）。
- **前置判断（while）と後置判断（do-while）の両方を実装する**。loop 構文は
  「条件位置（前置／後置）」をパラメータで切り替えられる **単一構文** とする。
  前置は条件をループ先頭に置き **ボディが 0 回** も実行され得る（stage 4-1）。
  後置はボディを **必ず 1 回** 実行してから判定する（後置検証ステージ、要件9）。
  両者は同じ部品（merge・cond・body・戻りエッジ・脱出経路）で構成され、
  **エッジの結線だけが異なる**（詳細は要件8）。
- ループボディは **通常スロットの一本道** とする。
- ランタイムの実行エンジン（`battleStore.startExecution`）は、現状すでに
  「ノードごとに次エッジを動的選択し、条件は到達のつど最新状態で再評価する」
  方式のため、閉路（ループ）をそのまま実行できる。本機能で実行エンジンに必要な
  追加は **無限ループ防止ガードのみ** という想定（詳細は設計で検証する）。

### 想定する記述イメージ（※具体的な JSON キーは設計フェーズで確定）

```jsonc
"4-1": {
  "enemyId": "wolf",
  "cards": [ { "id": "attack", "power": 15 }, { "id": "guard", "power": 10 } ],
  "flow": [
    {
      "loop": {
        "condition": "enemyHp == 0",
        "label": "てきのHPが0？",
        "trueDir":  "down",       // 成立(true)側＝脱出の出口方向
        "falseDir": "right",      // 不成立(false)側＝ボディへ進む出口方向
        "body": [
          {},
          { "lockedCard": { "id": "monster", "power": 20 } },
          {}
        ]
      }
    }
  ]
}
```

上記はあくまでイメージであり、キー名・構造は設計（`design.md`）で確定する。

> **方向指定は true / false（論理の枝）でキーする。** ループ専用の `exit` / `back`
> ではなく、通常の条件分岐ステージでも同じやり方で出口方向を指定できるようにする
> （要件2）。ただし既存の分岐ショートカット（3-1 / 3-2）では `true` / `false` キーが
> すでに **分岐ボディの配列** として使われているため、方向は別キー（例：`trueDir` /
> `falseDir`）か入れ子オブジェクトで表すなど、衝突を避ける形を設計フェーズで確定する。

## 要件

### 要件1: `flow` への loop 構文の追加とローダー展開
**ユーザーストーリー：** ステージデザイナーとして、`stages.json` の `flow` に while ループを
宣言的に書きたい。なぜなら、戻りエッジ・合流ノード・座標を手書きで組むのは間違えやすく、
ループステージの追加コストが高いから。

#### 受け入れ基準
1. WHEN `flow` 配列にループ要素（条件式とループボディを持つ要素）が含まれる THEN the system SHALL 合流ノード（merge）をループ条件ノードの手前に自動生成した完全形式（`slots` / `conditions` / `mergeNodes` / `edges` / `start` / `goal`）へ展開する
2. WHEN ループ要素を展開する THEN the system SHALL 「merge → cond」「cond →(不成立)→ ボディ先頭」「ボディ各スロットを順に連結」「ボディ末尾 → merge への戻りエッジ」を生成する
3. WHEN ループ要素を展開する THEN the system SHALL 条件成立側（脱出）の経路を、ループの後続要素（無ければ `goal`）へ接続する
4. WHEN ループボディの各スロットを展開する THEN the system SHALL 通常スロットと同様に連番採番し、`lockedCard` / `acceptOnly` / `multiplier` の既存オプションを取り込む
5. IF `flow` にループ要素が含まれない場合 WHEN ステージを展開する THEN the system SHALL 既存の線形展開・分岐展開（`processSubFlow` の auto-merge）を一切変更せず従来どおり動作させる
6. IF ループ要素の記述が不正（条件式が無い、ボディが配列でない等）の場合 WHEN 展開する THEN the system SHALL `console.warn` で警告を出し、アプリをクラッシュさせずに当該ループを安全側にスキップする

### 要件2: 条件ノードの true / false 出口方向の指定
**ユーザーストーリー：** ステージデザイナーとして、条件分岐の true / false がそれぞれどの方向
（上下左右）へ伸びるかを `stages.json` で指定したい。なぜなら while ループでは
「成立で下（Goal へ）」「不成立で右（ボディへ）」のように、既存の固定方向
（true=右 / false=下）と異なる配置が必要だから。

#### 受け入れ基準
1. WHEN 条件ノードに true / false の出口方向（`right` / `left` / `up` / `down`）が指定される THEN the system SHALL `ConditionNode` の対応するソースハンドルをその方向の辺に配置する
2. WHEN 出口方向の指定が省略される THEN the system SHALL 既存デフォルト（true=右 / false=下）を維持する
3. IF 出口方向が指定される THEN the system SHALL NOT 分岐の論理（どちらが true / false か、`selectNextEdge` が `sourceHandle` 名で選ぶ挙動）を変更しない（方向は純粋に視覚表現のみに影響する）
4. WHEN 同一方向に true と false の両方が指定される等の不正があった場合 THEN the system SHALL `console.warn` で警告し、デフォルト方向にフォールバックする

### 要件3: 合流ノード（merge）への戻りエッジの描画
**ユーザーストーリー：** プレイヤーとして、ループの繰り返しが「線が前に戻っている」と
目で見て分かってほしい。なぜなら、繰り返している実感がループ理解の核心だから。

#### 受け入れ基準
1. WHEN ループボディ末尾から条件手前の merge ノードへ戻るエッジを描画する THEN the system SHALL フローチャートの上側を通る経路で、merge ノードの上辺（または戻り用に割り当てたハンドル）へ接続する
2. WHEN merge ノードを描画する THEN the system SHALL 「start からの入力」「戻りエッジの入力」「cond への出力」を区別できるハンドル構成を提供する
3. WHILE 実行シーケンスが戻りエッジ・ループ内ノードを通過する the system SHALL 既存の通過演出（`traversedEdgeIds` / `traversedNodeIds` による発光トレイル）を周回ごとに反映する

### 要件4: ランタイムでのループ実行
**ユーザーストーリー：** プレイヤーとして、実行ボタンを押すと、条件が満たされるまで
ループボディが繰り返し実行され、満たされたら脱出してほしい。

#### 受け入れ基準
1. WHILE ループ継続中 the system SHALL ループボディ各ノードのカード効果（`attack` / `heal` / `guard` / `reflect` / `monster` 等）を毎周回適用し、戻りエッジを通って次の周回を継続する（結線は要件8の条件位置〔前置／後置〕に従う）
2. WHEN 実行シーケンスが条件ノードに到達する THEN the system SHALL その時点の最新状態（`enemyHp` / `playerHp` など）で条件を再評価する
3. WHEN 条件が成立する THEN the system SHALL ループを脱出し、脱出経路（`goal` 等）へ進む
4. IF ループ実行中にプレイヤーHPが 0 になる WHEN ボディのダメージ処理が走る THEN the system SHALL 既存の `failPhase` 機構でループを停止し Fail とする

### 要件5: 無限ループ防止ガード
**ユーザーストーリー：** 開発者／プレイヤーとして、条件が永遠に成立せず、かつプレイヤーも
死なない組み方をしたときに、実行が無限に続かないでほしい。なぜなら無限ループは画面が
固まり、子どもが「壊れた」と感じてしまうから。

#### 受け入れ基準
1. WHILE ループ実行中 the system SHALL 周回数（または総ノード実行ステップ数）をカウントする
2. IF 周回数が上限（**提案値: 100 周**、レビューで確定）を超える WHEN ループ実行中 THEN the system SHALL 実行を打ち切り、Fail オーバーレイを表示する
3. WHEN 上限超過で打ち切る THEN the system SHALL 進行中の実行タイマーをすべてキャンセルし、状態を整合させる（多重実行・幽霊タイマーが残らない）
4. the system SHALL この上限を「正常なステージでは到達しない安全弁」として扱い、通常の勝敗（敵HP 0／プレイヤーHP 0）が先に成立する場合はそちらを優先する

### 要件6: stage 4-1 の定義（具体的成果物）
**ユーザーストーリー：** プレイヤーとして、stage 4-1 で while ループのパズルを遊びたい。

#### 受け入れ基準
1. the system SHALL stage 4-1 を `start → merge → cond(敵HPが0？)` の構造で提供する
2. WHEN cond が不成立（敵HP ≠ 0） THEN the system SHALL 右方向のループボディ「空き → monster(power 20) → 空き」を実行し、上側を通って merge へ戻る
3. WHEN cond が成立（敵HP = 0） THEN the system SHALL 下方向の Goal へ進み、勝利判定へ移る
4. the system SHALL stage 4-1 の手札を 攻撃(power 15)・防御(power 10)、敵を HP=60 として定義する
5. WHEN stage 4-1 を `flow` の loop 構文で記述する THEN the system SHALL 戻りエッジ・merge・座標をデザイナーが手書きせずに展開できる（要件1 の達成確認を兼ねる）

> **未確定（設計／レビューで確定する事項）**
> - 4-1 の `enemyId`（既存スプライト `slime` / `wolf` / `knight` / `golem` から選定）。
> - 条件式の表記：`evaluateCondition` が `==` をサポートするか要確認。未対応なら
>   `enemyHp === 0` または `enemyHp <= 0`（オーバーキル時の取りこぼし防止）に調整する。
> - 4-1 の手札（攻撃15・防御10）と敵HP=60 で「クリア可能」かのバランスは、実装後の
>   プレイテストで微調整しうる（本要件は構造の固定であり、数値の最終確定ではない）。
> - 4-1 をプレイヤーへどう露出するか（`demoStageIds` 追加 or マップ進行解放）。

### 要件7: 未使用コードの整理
**ユーザーストーリー：** 開発者として、使われていない関数（`buildExecutionPath` 等）を
削除したい。なぜなら、ループ実行の実体は `startExecution` の動的なエッジ追跡であり、
`buildExecutionPath` のような静的経路展開は閉路（ループ）を扱えず、読む人を誤解させる
死にコードだから。

#### 受け入れ基準
1. WHEN 本機能の実装を行う THEN the system SHALL `battleStore.buildExecutionPath`（呼び出し元が存在しない死にコード）を削除する
2. WHEN `buildExecutionPath` を削除する THEN the system SHALL それを参照する古い docstring（実行シーケンスの説明文）を、実体（`startExecution` の動的エッジ追跡）に合わせて更新する
3. IF 本作業の周辺で他にも参照されていない関数・定数が見つかった場合 THEN the system SHALL grep で影響範囲を確認したうえで、確実に未使用なもののみ削除する

### 要件8: 条件位置パラメータ（前置 while / 後置 do-while）
**ユーザーストーリー：** ステージデザイナーとして、loop 構文で前置 while と後置 do-while を
切り替えたい。なぜなら、コース開発班の今後のコースで「条件を先に判定する繰り返し」と
「まず 1 回やってから判定する繰り返し」の両方を教える必要があるから。

#### 受け入れ基準
1. WHEN loop 構文に条件位置パラメータ（前置／後置）が指定される THEN the system SHALL 指定に応じて結線する。**前置**：`start → merge → cond`、cond 不成立 → ボディ先頭、ボディ末尾 → merge（戻り）、cond 成立 → 脱出。**後置**：`start → merge → ボディ先頭`、ボディ末尾 → cond、cond ループ側 → merge（戻り）、cond 脱出側 → 脱出
2. WHEN 条件位置パラメータが省略される THEN the system SHALL 前置（while）をデフォルトとする
3. the system SHALL 前置・後置のどちらでも、合流ノード（merge）・条件ノード（cond）・戻りエッジ・周回上限ガード（要件5）を同じ仕組みで共有し、差異をエッジの結線のみに閉じ込める
4. WHILE 後置ループを実行する the system SHALL ボディを最低 1 回実行してから条件を初回評価する
5. IF 条件位置パラメータが不正値（前置／後置以外）の場合 THEN the system SHALL `console.warn` で警告し、前置にフォールバックする

### 要件9: 後置 do-while の検証ステージ
**ユーザーストーリー：** 開発者として、後置 do-while が実際のステージで正しく動くことを
確認したい。なぜなら、構文だけ作っても描画・実行で破綻しないかは実ステージでしか
検証できないから。

#### 受け入れ基準
1. the system SHALL 後置 do-while を使う検証用ステージを 1 つ提供する（仮称 stage 4-2、番号はコース構成にあわせ調整可）
2. WHEN 検証ステージを実行する THEN the system SHALL ボディが最低 1 回実行され、条件成立で脱出することを目視・実行で確認できる
3. the system SHALL 検証ステージを前置の 4-1 と同じ `flow` の loop 構文（条件位置＝後置）で記述し、戻りエッジ・merge・座標を手書きせずに展開できることを示す

> **未確定（後置検証ステージの具体仕様、設計／レビューで確定）**
> 数値・敵・ボディ内容はストローマン。コース開発班の意図があれば差し替える。
> - ストローマン案：`enemyId=slime`、手札＝攻撃(20)・防御(10)、敵 HP=40、
>   ボディ＝「空き → monster(15) → 空き」、condition＝「てきのHPが0？」（後置）。
> - ステージ番号（4-2 とするか）と、プレイヤーへの露出方法（demo / マップ解放）。

## 対象外（今回のスコープ外）

- 入れ子ループ（ループの中にループ）。
- ループボディ内での条件分岐（cond のネスト）。
- `for`（回数指定）・`break` / `continue` 相当の構文。
  （`do-while`＝後置判断は今回スコープ **内**、要件8・9）
- ループ専用の特別な演出（周回カウンタ表示など）。既存の通過トレイル演出のみ流用する。

## 用語

| 用語 | 意味 |
|---|---|
| loop 構文 | `flow` 配列に書くループの宣言。ローダーが完全形式に展開する。条件位置で前置／後置を切り替える。 |
| 前置判断（while） | 条件をループ先頭で判定する。ボディは 0 回も実行され得る。 |
| 後置判断（do-while） | 条件をループ末尾（ボディの後）で判定する。ボディは必ず 1 回以上実行される。 |
| ループボディ | ループが継続するあいだ繰り返し実行される、通常スロットの一本道。 |
| 戻りエッジ | ループボディ末尾から、条件手前の merge ノードへ戻るエッジ。閉路を作る。 |
| 脱出経路 | 条件成立時に進む経路（`goal` 等）。 |
| 周回上限ガード | 無限ループを防ぐため、規定周回数を超えたら実行を打ち切る安全弁。 |
