# 要件定義: 敵のスプライトアニメーション描画

## はじめに
戦闘画面の上部エリアに敵をスプライトアニメーションで描画する機能。現状はスライムだけが存在するが、今後ボスや他の敵が追加されるため、敵の種類に依らず再利用できる汎用コンポーネントとして実装する。今回のスペックでは「idle 状態のループ再生」のみをスコープとし、将来的な状態別アニメーション（被弾・攻撃・死亡など）の追加が容易な設計にする。

## 要件

### 要件1: 敵スプライトの描画
**ユーザーストーリー：** プレイヤーとして、戦闘画面に敵の姿が表示されてほしい。なぜなら戦っている相手が誰なのかを視覚的に把握したいから。

#### 受け入れ基準
1. WHEN 戦闘画面が表示される THEN the system SHALL 敵スプライトを上部エリアの中央に 1 体描画する
2. WHEN 敵スプライトを描画する THEN the system SHALL 画像を原寸（そのままのピクセルサイズ）で表示する
3. IF 敵の画像が読み込めない WHEN 戦闘画面が表示される THEN the system SHALL 他の UI 要素（HP・フローチャート等）のレイアウトを崩さない

### 要件2: idle アニメーションのループ再生
**ユーザーストーリー：** プレイヤーとして、敵が静止画ではなくアニメーションで動いていてほしい。なぜなら戦闘画面に生きている感じが欲しいから。

#### 受け入れ基準
1. WHEN 戦闘画面が表示される THEN the system SHALL idle アニメーションを自動で開始する
2. WHILE idle アニメーションを再生している the system SHALL フレーム画像を一定間隔で順次切り替える
3. WHEN 最終フレームを表示し終えた THEN the system SHALL 先頭フレームに戻ってループを継続する
4. WHERE 戦闘画面が表示されている間 the system SHALL アニメーションを停止せず再生し続ける

### 要件3: 敵種類の汎用化
**ユーザーストーリー：** 開発者として、新しい敵を追加するときに共通コンポーネントへデータ（敵 ID・フレーム情報など）を渡すだけで描画できるようにしたい。なぜなら敵ごとに描画コードを書き直すのは手間で、バグの温床になるから。

#### 受け入れ基準
1. WHEN 新しい敵を追加する THEN the system SHALL 敵識別子（例：`slime`）と状態（例：`idle`）を指定するだけでスプライトを解決できるようにする
2. IF 敵ごとに総フレーム数が異なる WHEN アニメーションを再生する THEN the system SHALL その敵の定義に従って正しい枚数のフレームを循環させる
3. WHEN 敵の定義を JSON 等のデータとして管理する THEN the system SHALL コード変更なしで敵の追加・フレーム数変更ができる

### 要件4: 状態別アニメーションへの拡張性
**ユーザーストーリー：** 開発者として、将来 idle 以外の状態（被弾・攻撃・死亡など）を追加するときに既存コードを大きく書き換えずに済むようにしたい。なぜなら後から機能追加するときの影響範囲を最小にしたいから。

#### 受け入れ基準
1. WHEN 状態別アニメーションを設計する THEN the system SHALL 状態名（`idle`/`hurt`/`attack` など）をキーとしてフレーム群を切り替えられるデータ構造にする
2. IF 将来新しい状態が追加される WHEN 敵コンポーネントを利用する THEN the system SHALL 現行の呼び出し箇所（idle のみを使う箇所）を変更せずに新状態を追加できる
3. WHERE 今回のスペック the system SHALL idle 状態のみ実装し、他の状態は実装しない（ただしデータ構造・型定義は状態拡張を前提とする）

### 要件5: ディレクトリ・命名規則への準拠
**ユーザーストーリー：** 開発者として、敵スプライトの置き場所と命名が一貫していてほしい。なぜなら規約がバラバラだと管理しづらく、JSON からの参照もエラーが起きやすいから。

#### 受け入れ基準
1. WHEN 敵スプライトを配置する THEN the system SHALL `frontend/public/sprites/enemies/<敵ID>/<状態>/` 配下に格納する
2. WHEN 敵スプライトファイルを命名する THEN the system SHALL `<敵ID>_<状態>_<2桁ゼロ埋め連番>.png` の形式（例：`slime_idle_00.png`）に従う
3. WHERE 連番は the system SHALL 0 始まり・最低 2 桁のゼロ埋めとし、フレーム数が 100 以上になる場合のみ 3 桁に拡張する
4. WHEN ディレクトリを追加・変更する THEN the system SHALL 同じコミット内で `README.md` のディレクトリ構造セクションを更新する
